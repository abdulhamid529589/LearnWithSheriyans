Local Storage, Session Storage, and Cookies in JavaScript

Modern web development often requires client-side persistence: storing and retrieving data in the browser, across sessions or tabs, or just for temporary state. The three main tools for this are localStorage, sessionStorage, and cookies.

1. localStorage

localStorage is a key-value storage mechanism available in almost all browsers. Data stored here:

  • Persists even after closing the browser/tab.
  • Is only accessible by scripts from the same origin.
  • Is NOT sent to the server on each HTTP request.
  • Limit: ~5-10 MB (way more than cookies).

// Save data
localStorage.setItem('theme', 'dark');
// Load data
const theme = localStorage.getItem('theme');
// Remove data
localStorage.removeItem('theme');
// Clear all
localStorage.clear();

Best for: User preferences (theme, language), app state, offline data, shopping carts, drafts, caching API data, etc.

2. sessionStorage

sessionStorage is nearly identical to localStorage except:

  • Data only persists as long as the tab or window is open.
  • Closing the tab/window wipes it out.
  • Different tabs do not share sessionStorage, even for the same site.

// Save data
sessionStorage.setItem('sessionID', '12345');
// Load data
const sessionID = sessionStorage.getItem('sessionID');
// Remove data
sessionStorage.removeItem('sessionID');

Best for: Temporary state, forms data (preventing loss on accidental navigation), wizards/steps, temporary tokens, etc.

3. Cookies (document.cookie)

Cookies are name-value pairs stored both in the browser and (optionally) sent with every request to a server (for this origin/path/domain). Unique features:

  • Automatically included in HTTP(S) requests by the browser (unlike localStorage/sessionStorage).
  • Small size (~4KB per cookie; number and total size limits per domain).
  • Attributes: expires (expiry date), max-age, secure, SameSite, path, HttpOnly*.
  • HttpOnly cookies not accessible via JS (good for security for session/auth tokens).
  • Best for: authentication/session IDs, analytics, tracking, remembering login status, cross-page/server data.
  • Must use document.cookie API (awkward!) for set/read via JS.

// Set a cookie (expires in 7 days)
document.cookie = "userId=abc123; path=/; max-age=" + 7*24*60*60;
// Read all (as a string, must parse)
console.log(document.cookie); // "userId=abc123; theme=dark"
// JS cannot set HttpOnly cookies

Reminder: Do not store sensitive data in localStorage/sessionStorage as JS is always accessible to XSS. Use cookies with HttpOnly and Secure flags for server-side auth session ids.

Best Real-World Project Example: Powerful Login, Theme, and Cart System

Imagine a large e-commerce web app:

  • Authentication/Session (Cookies): When users log in, server sets a Set-Cookie: sessionid=... header. App uses HttpOnly + Secure + SameSite cookies for session management. User stays logged in across all tabs and page reloads because browser sends the cookie with every request.
  • User Preferences (localStorage): User picks "dark theme". App stores localStorage.setItem('theme', 'dark'). On every load, JS reads it and updates UI instantly, even if user returns days later.
  • Shopping Cart (localStorage or sessionStorage): The cart contents are written to localStorage (so they persist forever if user is not logged in) or sessionStorage (if you want the cart wiped when tab closes). This way, items remain in the cart if the user closes and reopens the browser/tab.
Example scenario:
- You visit shop.com and add items to the cart. Cart lives in localStorage.
- You choose "dark mode"; it's stored in localStorage.
- You log in; server sets sessionid in an HttpOnly, Secure cookie.
- You reload/come back days later: cart and theme preference remain, and if your session cookie is still valid, you are logged in.

Tips & Best Practices
  • Avoid storing sensitive info (tokens, passwords, PII) in localStorage/sessionStorage!
  • Always use HttpOnly, Secure, SameSite flags for auth cookies (set them from server-side).
  • Keep cookies as small as possible; they slow down every user request.
  • For larger or more complex client-side data, use localStorage or IndexedDB (for huge/structured data).
  • Consider how your app should behave for privacy (GDPR!): get consent for cookies & storage if needed.
  • Remember: closing all tabs ends sessionStorage, reloading the page does not.
